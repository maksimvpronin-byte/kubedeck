# KubeDeck 2.4.3 — Cluster Namespaces and Drawer Selection Plan

Статус: реализация, автоматические проверки и macOS packaging завершены; ручной UI smoke и Windows smoke отложены.

## Цель

Исправить два связанных дефекта навигации:

1. Хранить выбранные namespaces отдельно для каждого кластера, чтобы переключение кластера никогда не отправляло запросы в namespaces предыдущего кластера.
2. Сразу показывать в resource drawer данные выбранного объекта, без сохранения информации от предыдущего ресурса и без необходимости вручную переключать вкладку.

Версия релиза: `2.4.3`.

## Подтверждённые замечания

### 1. Общий namespace между кластерами

Сейчас `selectedNamespaces` является общей выборкой приложения. При переходе из cluster A в cluster B она продолжает содержать namespaces cluster A. Если таких namespaces нет в cluster B, таблица запрашивает неверный scope и остаётся пустой.

Дополнительно `lastNamespacedSelectionRef` в `useResourceNavigation` хранит только одну последнюю выборку. Поэтому переход через cluster-scoped ресурсы (`Nodes`, `Namespaces`, CRD definitions и другие) также может восстановить namespaces другого кластера.

### 2. Несогласованная выбранная строка drawer

В `App` строка (`selectedPod`) и её тип (`selectedResource`) хранятся двумя независимыми состояниями. При переходе, например, с Secret на Pod drawer может некоторое время получить несогласованную пару: новую строку со старым resource либо старую строку с новым resource.

Lifecycle drawer уже использует identity `cluster/resource/namespace/name/uid` и защищён от части stale responses, однако смена выбранного объекта должна происходить атомарно во всём пути: таблица → `App` → drawer → вкладки.

Это предварительное объяснение дефекта. Во время реализации его нужно подтвердить автоматическим regression-тестом и проверкой конкретного сценария Secret → Pod.

## Границы патча

Входит:

- отдельная namespaced-выборка для каждого `clusterId`;
- поддержка одиночного и множественного выбора namespaces;
- восстановление выбора при возврате в уже посещённый кластер;
- безопасный fallback для нового кластера и удалённых namespaces;
- сохранение выбора при переходах между namespaced-ресурсами;
- сохранение namespaced-выбора при временном переходе на cluster-scoped ресурс;
- атомарная выбранная цель drawer: resource + row;
- немедленный reset содержимого при реальной смене объекта;
- защита от старых async-ответов предыдущего объекта/кластера;
- автоматические контракты и версия `2.4.3`.

Не входит:

- синхронизация namespace-настроек между разными компьютерами;
- изменение Kubernetes API, Gateway или kubectl-команд;
- изменение частоты auto-refresh;
- запоминание отдельного namespace для каждого типа ресурса внутри одного кластера;
- полный редизайн Namespace Selector или resource drawer;
- Windows smoke до отдельной проверки на Windows-машине.

## Критерий готовности

Каждый кластер независимо запоминает свой одиночный или множественный выбор namespaces. Переключение кластеров никогда не использует namespaces предыдущего кластера, а переход через cluster-scoped ресурсы не уничтожает сохранённый namespaced-выбор.

При выборе другого Kubernetes-объекта drawer сразу и согласованно показывает его header, Summary, actions и данные активной вкладки. Информация предыдущего объекта или кластера не остаётся на экране и не может вернуться из позднего async-ответа.
