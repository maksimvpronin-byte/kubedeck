# KubeDeck 1.0.4 Release Specification

Дата фиксации спецификации: 2026-06-03

## 1. Назначение релиза

KubeDeck 1.0.4 — релиз, сфокусированный на операциях с Kubernetes Nodes и SSH-доступе внутрь нод из интерфейса приложения.

Helm management был рассмотрен как кандидат для 1.0.4, но исключён из scope релиза, чтобы не раздувать патч. Helm переносится в отдельный будущий релиз.

## 2. Scope релиза 1.0.4

В релиз входят:

1. Операции над нодами:
   - cordon;
   - uncordon;
   - drain;
   - bulk-операции по нескольким выбранным нодам.
2. Preview затрагиваемых Pod перед drain.
3. SSH-доступ внутрь ноды из Node drawer.
4. SSH default settings на уровне приложения/кластера без хранения секретов.
5. Интеграция node/SSH операций с существующим action/operations-разделом приложения.
6. Bugfixes, если они всплывут во время реализации 1.0.4.
7. Поднятие версии проекта до 1.0.4.
8. Git tag `1.0.4` после финальной проверки релиза.

## 3. Out of scope для 1.0.4

В 1.0.4 не реализуется:

1. Helm releases/charts/repositories management.
2. Установка Helm charts.
3. Helm upgrade/rollback/uninstall.
4. Хранение SSH-паролей.
5. Хранение приватных SSH-ключей.
6. Хранение Helm repository credentials.

## 4. Версионирование и Git

К финальному патчу 1.0.4 версия должна быть поднята во всех релевантных местах проекта:

- root `package.json`;
- desktop package metadata, если версия дублируется там;
- electron-builder output name;
- README;
- CHANGELOG;
- validation/package scripts, если они содержат `1.0.3` в имени или тексте;
- release notes.

Финальный Git tag:

```powershell
git tag -a 1.0.4 -m "KubeDeck 1.0.4"
git push origin main
git push origin 1.0.4
```

Tag создаётся только после успешной ручной проверки portable-сборки.

## 5. Node UI

### 5.1 Размещение

Использовать уже существующее место размещения Nodes в проекте. Новый раздел Nodes с нуля не создавать, если текущая структура уже содержит Nodes page/table.

### 5.2 Таблица Nodes

Использовать текущие колонки Nodes из проекта. В рамках первого патча node actions не менять модель колонок без необходимости.

Если по ходу реализации потребуется минимальное расширение, допустимо добавить только технически нужные поля:

- `metadata.name`;
- `spec.unschedulable`;
- `status.addresses` для InternalIP;
- `status.conditions` для Ready/NotReady;
- `status.nodeInfo` при уже существующем отображении.

### 5.3 Multi-select

На экране списка Nodes должна быть возможность выбрать одну или несколько нод.

Для выбранных нод показывать bulk actions:

- Cordon selected;
- Uncordon selected;
- Drain selected;
- Clear selection.

Для одной выбранной ноды эти же действия доступны из строки/контекстного меню/Node drawer, если соответствующие UI-механики уже есть в проекте.

## 6. Cordon / Uncordon

### 6.1 Команды

Cordon:

```powershell
kubectl cordon <node>
```

Uncordon:

```powershell
kubectl uncordon <node>
```

Команды должны выполняться через уже существующий механизм запуска `kubectl` проекта, с тем же kubeconfig/current context, который выбран в UI.

### 6.2 Bulk behavior

Если выбрано несколько нод, операция выполняется по каждой ноде отдельно.

Требуемое поведение:

1. Модальное окно подтверждения закрывается сразу после Confirm.
2. Операция уходит в background/action tracker.
3. Для каждой ноды показывается отдельный результат:
   - pending;
   - running;
   - success;
   - failed.
4. Ошибка на одной ноде не должна останавливать обработку остальных нод.
5. После завершения список Nodes обновляется.

### 6.3 Confirmation

Для cordon/uncordon использовать простой confirm по аналогии с уже реализованным удалением Pod.

Typed confirmation для 1.0.4 не требуется.

## 7. Drain

### 7.1 Команда по умолчанию

Для drain использовать фиксированную команду без дополнительных чекбоксов:

```powershell
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data --timeout=300s
```

### 7.2 Confirmation

Для drain использовать простой confirm по аналогии с уже реализованным удалением Pod.

Typed confirmation для 1.0.4 не требуется.

### 7.3 Drain preview

Перед запуском drain показывать preview Pod, которые находятся на выбранных нодах.

Минимальный набор полей preview:

- node name;
- namespace;
- pod name;
- pod phase/status;
- owner kind/name, если легко получить из `ownerReferences`;
- restart count, если поле уже доступно в текущей модели Pod;
- age, если поле уже доступно в текущей модели Pod.

Preview должен показывать все Pod на выбранных нодах, сгруппированные по node name.

DaemonSet-managed Pod можно показывать в том же списке, но с пометкой `DaemonSet`, потому что `--ignore-daemonsets` означает, что drain не будет удалять DaemonSet-managed Pod.

### 7.4 Bulk drain behavior

Если выбрано несколько нод:

1. Сначала показывается общий preview по всем выбранным нодам.
2. После Confirm создаётся background operation.
3. Drain выполняется по нодам последовательно, а не параллельно.
4. По каждой ноде фиксируется отдельный статус.
5. Ошибка на одной ноде не должна скрывать результаты остальных.
6. После завершения обновляется список Nodes и Events.

Последовательный drain выбран как более безопасный вариант, чтобы не эвакуировать слишком много Pod одновременно.

## 8. RBAC preflight / `kubectl auth can-i`

`kubectl auth can-i` нужен не для безопасности сам по себе, а для ранней диагностики прав пользователя до запуска опасной операции.

Назначение preflight:

1. До запуска операции показать понятную ошибку: нет прав на patch nodes / evict pods / delete pods.
2. Не отправлять пользователя в непонятную ошибку `Forbidden` после запуска долгой операции.
3. Позволить UI заранее отключить действие или показать warning.
4. Уменьшить риск ситуации, когда операция частично стартовала и затем упала из-за RBAC.

Для 1.0.4 решение:

- preflight можно реализовать как warning/diagnostic;
- если `can-i` явно вернул `no`, показывать понятное предупреждение;
- если `can-i` сам не сработал или недоступен, не блокировать UI полностью, а дать пользователю запустить команду и увидеть реальный вывод `kubectl`;
- финальный источник истины — результат реальной команды `kubectl`.

Потенциальные проверки:

```powershell
kubectl auth can-i patch nodes
kubectl auth can-i create pods/eviction -A
kubectl auth can-i delete pods -A
```

## 9. SSH into Node

### 9.1 Архитектура

SSH реализуется через библиотеку внутри приложения, а не через системный `ssh.exe`.

Предпочтительная реализация:

- Electron main process держит SSH-сессию;
- UI использует существующий терминальный компонент или xterm.js;
- обмен идёт через IPC/WebSocket-подобный stream внутри desktop-приложения;
- Python backend не должен хранить SSH-секреты;
- пароль и passphrase живут только в памяти текущей сессии.

### 9.2 Размещение в UI

SSH открывается в Node drawer.

В Node drawer добавить секцию:

- SSH connection form;
- Connect button;
- embedded terminal;
- Disconnect button;
- connection status;
- last error.

### 9.3 Адрес ноды

Адрес по умолчанию: `InternalIP` из `Node.status.addresses`.

Если InternalIP отсутствует, fallback:

1. Hostname;
2. ExternalIP;
3. ручной ввод host/IP в форме SSH.

### 9.4 Auth modes

В 1.0.4 поддержать следующие способы подключения:

1. Username + password.
2. Username + private key file selected from disk.
3. Username + private key file + passphrase.
4. Username + SSH agent/default local SSH identity, если это технически доступно в выбранной библиотеке.
5. Jump host/bastion.

### 9.5 Jump host

Добавить возможность подключения через jump host.

Минимальные поля:

- enable jump host;
- jump host address;
- jump host port;
- jump username;
- jump auth mode;
- jump password/passphrase only in memory;
- target host remains node InternalIP by default.

### 9.6 Хранение настроек SSH

Разрешено хранить в `%APPDATA%\KubeDeck\config.json` только несекретные значения:

```json
{
  "ssh": {
    "defaultUser": "",
    "defaultPort": 22,
    "preferAddressType": "InternalIP",
    "jumpHost": {
      "enabled": false,
      "host": "",
      "port": 22,
      "username": ""
    }
  }
}
```

Запрещено хранить:

- SSH password;
- SSH private key content;
- SSH private key passphrase;
- Helm repo credentials;
- любые токены/секреты.

Путь к private key в 1.0.4 по умолчанию не сохранять. Его можно использовать только в текущей форме подключения.

## 10. Action / Operations integration

Все операции 1.0.4 должны попадать в существующий action/operations-раздел, если он уже есть в проекте.

Типы операций:

- node.cordon;
- node.uncordon;
- node.drain;
- node.ssh.connect;
- node.ssh.disconnect.

Для каждой операции фиксировать:

- action id;
- cluster/context;
- node name;
- operation type;
- started at;
- finished at;
- status;
- command preview для kubectl-операций;
- stdout/stderr или краткую ошибку;
- affected resources count, если применимо.

Модальные окна не должны зависать в состоянии `Running`/`Deleting`/`Draining`. Confirm закрывает окно, выполнение продолжается в operations panel.

## 11. Error handling

### 11.1 Node actions

Ошибки показывать человекочитаемо:

- `Forbidden` / RBAC issue;
- node not found;
- timeout;
- drain blocked by PodDisruptionBudget;
- drain blocked by unmanaged Pod;
- kubectl executable not found;
- kubeconfig/current context unavailable.

### 11.2 SSH

Ошибки SSH показывать в Node drawer:

- host unreachable;
- connection timeout;
- authentication failed;
- private key read error;
- passphrase required;
- jump host connection failed;
- target connection through jump host failed;
- host key verification issue, если библиотека это возвращает.

## 12. Security requirements

1. Не хранить пароли.
2. Не хранить приватные ключи.
3. Не логировать пароли/passphrase/private key material.
4. Не писать SSH password/passphrase в desktop/backend/kubectl logs.
5. Для команд kubectl логировать command preview без секретов.
6. Для SSH логировать только host/user/port/status без паролей и ключей.
7. Не выполнять drain параллельно по нескольким нодам.
8. После SSH disconnect закрывать stream и очищать секреты из памяти насколько это возможно на уровне JS runtime.

## 14. Тестирование

### 14.1 Автоматические проверки

Перед выдачей каждого code patch по возможности выполнять локально:

```powershell
npm.cmd run build
py -3 -m pytest .\apps\backend\tests
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\validate-1.0.3.ps1
```

После version bump скрипт проверки должен быть обновлён на 1.0.4, если файл будет переименован.

### 14.2 Manual test checklist

Node actions:

1. Открыть Nodes.
2. Выбрать одну ноду.
3. Выполнить cordon.
4. Проверить, что нода стала unschedulable.
5. Выполнить uncordon.
6. Проверить, что unschedulable снят.
7. Выбрать несколько нод.
8. Проверить bulk cordon/uncordon.
9. Перед drain увидеть preview Pod с namespace.
10. Выполнить drain на тестовой ноде.
11. Проверить operation result.
12. Проверить, что UI не зависает в modal running state.

SSH:

1. Открыть Node drawer.
2. Проверить, что host по умолчанию = InternalIP.
3. Подключиться по username/password.
4. Подключиться по private key file.
5. Подключиться через jump host.
6. Проверить disconnect.
7. Проверить, что пароль/passphrase не появились в логах.

## 15. Acceptance criteria релиза 1.0.4

Релиз считается готовым, если:

1. Версия проекта поднята до 1.0.4.
2. Nodes page поддерживает выбор нескольких нод.
3. Cordon работает для одной и нескольких нод.
4. Uncordon работает для одной и нескольких нод.
5. Drain показывает preview Pod по namespace и node.
6. Drain выполняется фиксированной командой:

```powershell
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data --timeout=300s
```

7. Drain по нескольким нодам идёт последовательно.
8. Все node actions отображаются в operations/action разделе.
9. SSH открывается в Node drawer.
10. SSH поддерживает password auth и private key auth.
11. SSH jump host реализован.
12. Пароли и ключи не сохраняются.
13. Portable build собирается.
14. README и CHANGELOG обновлены.
15. Git tag `1.0.4` создан после успешной проверки.

## Final 1.0.4 scope update

The 1.0.4 release scope was finalized without Helm support. Helm chart management is explicitly deferred to a later release.

Implemented release scope:

- Node bulk actions: `cordon`, `uncordon`.
- Node drain with pod preview and fixed command: `kubectl drain <node> --ignore-daemonsets --delete-emptydir-data --timeout=300s`.
- In-app SSH from the Node drawer using the backend SSH implementation.
- SSH defaults in Settings for non-secret values only.
- Confirmation modals use KubeDeck UI instead of native browser/Windows dialogs.

Security constraints:

- SSH passwords are not persisted.
- SSH passphrases are not persisted.
- SSH private key paths are not persisted as defaults.
- Helm is not part of 1.0.4.
